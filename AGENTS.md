# AGENTS.md

このファイルは、このリポジトリで作業するエージェント向けの開発・検証・配布ガイドです。READMEや過去の説明よりも、現在のコード、テスト、ビルドスクリプトを優先してください。

## プロジェクト概要

- アプリ名はMovie Downloaderです。
- Windows 10/11 x64向けのC# / WPFアプリケーションです。
- ターゲットは`.NET 10`の`net10.0-windows`です。
- 旧Python / Tkinter版は削除済みです。Python、PyInstaller、`main.py`を前提とした実装や配布手順を復元しないでください。
- アプリは単一URLから動画または音声をダウンロードします。複数URLキュー、プレイリスト管理、履歴は現在の対象外です。

## 参照の優先順位

仕様や実装状態を判断するときは、次の順に確認してください。

1. `src/YtdlGUI.Wpf`の実装
2. `tests/YtdlGUI.Wpf.Tests`のテスト
3. `build-wpf.ps1`と`setup-tools.ps1`
4. `README.md`、`THIRD_PARTY_NOTICES.md`、`design-qa.md`
5. Git履歴や過去の説明

ドキュメントとコードが食い違う場合は、実装とテストを確認したうえで、依頼範囲にドキュメント更新を含めてください。

## 開発環境

- 開発環境はWindowsとPowerShellを基準にしてください。
- `global.json`が.NET SDK `10.0.301`を指定しています。
- Visual Studioは必須ではありません。.NET 10 SDKがあればPowerShellからビルドできます。
- この環境では、`build-wpf.ps1`が`%USERPROFILE%\.dotnet\dotnet.exe`を優先し、見つからない場合だけPATH上の`dotnet`を使用します。

通常の検証と発行には、次のコマンドを使用してください。

```powershell
.\build-wpf.ps1
```

このスクリプトはRelease構成でテストを実行した後、自己完結型・単一ファイルのWindows x64版を`artifacts\publish\win-x64`へ発行します。発行先は、外部GPLバイナリの残留を避けるため、発行前に空になります。

テストだけを実行する場合は、次のコマンドを使用してください。

```powershell
& "$env:USERPROFILE\.dotnet\dotnet.exe" test YtdlGUI.slnx -c Release
```

ローカル実行に必要な外部ツールは、アプリ本体を発行した後に取得してください。

```powershell
.\setup-tools.ps1
```

`setup-tools.ps1`はネットワークへ接続し、外部配布元から実行ファイルを取得します。ビルドや単体テストだけが目的の場合は実行しないでください。

## 改行コード

- リポジトリ内のテキストファイルはLFを使用してください。
- Gitの改行判定は`.gitattributes`を正本とし、個人環境の`core.autocrlf`には依存しないでください。
- エディターの保存設定は`.editorconfig`に従ってください。
- CRLFとLFの違いだけで全行が変更された差分はコミットしないでください。
- PNGやICOなどのバイナリファイルには改行変換を適用しないでください。

## プロジェクト構成と責務

- `MainWindow.xaml`は画面構造とデータバインディングを担当します。
- `Resources/Styles.xaml`は色、余白、角丸などの共通スタイルを担当します。
- `MainViewModel`は画面状態、入力検証、コマンド、設定保存、非同期処理の調整を担当します。
- `IYtDlpService`はメタデータ取得、ダウンロード、バージョン確認、手動更新を抽象化します。
- `YtDlpService`は`yt-dlp.exe`の外部プロセス実行とキャンセルを担当します。
- `YtDlpCommandBuilder`は引数生成を担当します。
- `ProgressParser`はyt-dlpの進捗出力を`DownloadProgress`へ変換します。
- `JsonSettingsStore`は設定保存と旧`config.ini`からの初回移行を担当します。
- `Models/DownloadModels.cs`はプリセット、リクエスト、進捗、メタデータ、設定を定義します。

画面のコードビハインドやUIスレッドから、yt-dlpやffmpegのプロセスを直接操作しないでください。外部処理はサービス層へ置き、`Task`と`CancellationToken`を使って非同期に実行してください。プロセス引数は文字列連結ではなく`ProcessStartInfo.ArgumentList`へ個別に追加し、URLやパスに空白や特殊文字が含まれても崩れないようにしてください。

サービスやモデルのインターフェースを変更するときは、ViewModel、実装、テストを同じ変更で整合させてください。UIへ通知する進捗は`IProgress<DownloadProgress>`を通し、ダウンロード中に画面をブロックしないでください。

## 維持する動作仕様

- URLはHTTPまたはHTTPSだけを受け付けます。
- URL入力後、短い待機を挟んでメタデータを非同期取得します。
- 初期版では`--no-playlist`を維持し、単一動画だけを処理します。
- 動画プリセットはH.264を優先し、MP4へマージします。
- 音声プリセットはM4AとMP3を提供し、詳細設定からWAVへ変換できます。
- 出力ファイル名は`%(upload_date)s-%(title)s.%(ext)s`を維持します。
- サムネイル埋め込みは初期状態で有効です。ただし、yt-dlpがWAVへの埋め込みをサポートしないため、WAV変換時は埋め込み引数を付けないでください。
- 動画では日本語・英語字幕の取得と埋め込みを選択できます。
- ダウンロード中は進捗率、速度、残り時間、状態、キャンセルを表示します。
- ダウンロード処理には10分のタイムアウトがあり、キャンセル時は子プロセスも終了します。
- 設定は`%AppData%\YtdlGUI\settings.json`へ保存します。
- 同じフォルダに旧`config.ini`がある場合は、初回だけ有効な保存先を移行します。
- 起動時にはyt-dlpの更新有無だけを非同期確認し、自動更新はしません。更新は利用者の明示操作で実行します。
- エラーは画面内のメッセージと詳細ログへ伝え、アプリを操作可能な状態へ戻します。

これらを変更する場合は、意図的な仕様変更としてテストとREADMEも更新してください。

## UI方針

- WindowsのFluent Designに寄せた、情報量を抑えたミニマルUIを維持してください。
- Material Design系の大型テーマライブラリは導入せず、WPF標準コントロールと小さな独自XAMLスタイルを使用してください。
- 通常時に目立たせる主操作は、URL、形式、保存先、ダウンロードです。
- 字幕、サムネイル、WAV、ログ、更新操作などは補助領域や折りたたみ領域へ置いてください。
- Segoe UI、白から薄灰の背景、単一のアクセント色、8px前後の角丸、控えめな影を基本にしてください。
- システムのライト・ダークテーマ、キーボード操作、明確なフォーカス表示、十分なコントラスト、ウィンドウのリサイズ、200%表示を考慮してください。
- 装飾目的だけのアイコンや依存ライブラリを増やさず、操作の意味が明確なラベルを優先してください。

UIを変更した場合は、ライトテーマとダークテーマ、最小ウィンドウサイズ、長い実データ、キーボード移動、文字切れ、重なりを確認してください。視覚検証の内容や基準を変更した場合は`design-qa.md`も更新してください。

## 外部ツールとライセンス

- アプリ本体のライセンスはルートの`LICENSE`にあるMIT Licenseです。
- `yt-dlp.exe`、`ffmpeg.exe`、`ffprobe.exe`はリポジトリ、コミット、標準配布物へ含めないでください。
- `setup-tools.ps1`は、yt-dlpの公式GitHub ReleasesとGyanのFFmpeg配布元から、ローカル実行用のファイルを直接取得します。
- 公式のyt-dlp WindowsバイナリにはGPLv3+の第三者コードが含まれます。
- 現在取得するGyanのFFmpeg Essentials BuildはGPLv3です。
- 外部実行ファイルを含むフォルダを再配布する場合は、GPLのライセンス表示や対応するソースコードの提供義務を別途満たす必要があります。通常の発行処理では再配布しないでください。
- 自己完結型の発行物には.NETランタイムが含まれます。`THIRD_PARTY_NOTICES.md`と`licenses/DOTNET_RUNTIME_LICENSE.txt`を発行物から削除しないでください。

外部ツールの取得元、種類、同梱方針を変更するときは、実装前に各配布元の最新ライセンスを一次資料で確認し、`THIRD_PARTY_NOTICES.md`とREADMEを更新してください。

## テストと完了条件

少なくとも次の領域を自動テストで保護してください。

- MP4/H.264優先、M4A、MP3、WAV、字幕、サムネイル、出力テンプレートのコマンド生成
- 進捗率、速度、残り時間の解析
- 設定の保存、再読込、旧`config.ini`からの移行
- URL、保存先、外部ツール不足、終了コード異常、キャンセル、タイムアウトの復帰動作

変更後は、変更内容に応じて次を実行してください。

1. 関連する単体テストを実行する。
2. 原則として`.\build-wpf.ps1`を実行する。
3. `artifacts\publish\win-x64`に`yt-dlp.exe`、`ffmpeg.exe`、`ffprobe.exe`が混入していないことを確認する。
4. `git diff --check`で空白エラーを確認する。
5. `git status`と差分を確認し、依頼と無関係なファイルを含めない。
6. UI変更時は、実際の画面と操作を確認する。

コマンドの終了コードだけで完了と判断しないでください。発行物、対象ファイル、テスト件数、画面、PR差分など、依頼の中心となる実物を確認してから報告してください。環境上の理由で実行できなかった検証は、未確認として明記してください。

## Git運用

- 開発の基点は`develop`です。
- 機能開発は`develop`から`feature/<feature-name>`を作成してください。
- Codex専用の名前ではなく、内容を表すfeatureブランチ名を使用してください。
- マージは`--no-ff`相当のマージコミットを基本とします。
- PRのベースは`develop`にしてください。
- コミットメッセージは1行で簡潔にし、`feature:`、`fix:`、`refactor:`、`docs:`を変更内容に応じて使用してください。
- コミットメッセージとPRには絵文字を使用しないでください。
- 作業ツリーに既存変更がある場合は、その変更を利用者の作業として扱い、依頼と無関係なファイルを上書き、削除、ステージしないでください。
- ステージ時は対象ファイルを明示し、混在した作業ツリーで安易に`git add -A`を使用しないでください。
- `git reset --hard`や`git checkout --`などの破壊的な復元操作を使用しないでください。

## コミットしないもの

- `artifacts/`、`bin/`、`obj/`などの生成物
- `yt-dlp.exe`、`ffmpeg.exe`、`ffprobe.exe`、`main.exe`などの外部または旧実行ファイル
- `%AppData%\YtdlGUI\settings.json`や`config.ini`などの利用者設定
- ログ、一時ファイル、IDE固有設定、認証情報、秘密鍵

新しい生成物やローカル専用ファイルを導入した場合は、必要に応じて`.gitignore`を更新してください。ただし、追跡対象の成果物を隠すために無視設定を追加しないでください。
